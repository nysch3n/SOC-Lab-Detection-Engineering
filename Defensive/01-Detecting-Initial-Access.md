# 🛡️ Detection Analysis: Recon & Brute-Force

This document analyzes the SIEM (Splunk) telemetry generated during the attacks outlined in `Offensive/01-Recon-and-Bruteforce.md`.

## 1. The Reconnaissance Blind Spot (Nmap)
* **Observation:** The aggressive Nmap scan executed by the attacker generated **zero** actionable alerts in Splunk via Sysmon `EventCode=3` (Network Connection).
* **Root Cause Analysis:** 1. **Sysmon Architecture:** Sysmon monitors *processes*, not raw network packets. If Nmap scans a closed port, no Windows process intercepts the connection, so Sysmon does not log it.
    2. **Configuration Tuning:** The SwiftOnSecurity Sysmon configuration is heavily tuned to drop noisy inbound network connections to prevent SIEM resource exhaustion.
    3. **Windows Firewall:** The Domain Profile firewall dropped unauthorized inbound packets before they could trigger an application layer event.
* **Detection Takeaway:** Relying solely on endpoint EDR/Sysmon for network scanning detection is insufficient. Perimeter firewalls or Network Intrusion Detection Systems (NIDS) like Zeek/Suricata are required for this phase of the kill chain.

## 2. Detecting the SMB Brute-Force (Event ID 4625)
* **Objective:** Detect the `NetExec` dictionary attack against the `jkowalski` account.

### The Localization Engineering Challenge
During the analysis of **Event ID 4625 (Logon Failure)**, a critical real-world engineering challenge was encountered. The target Windows OS was localized in **Polish**. Consequently, standard Splunk parsers (Splunk Add-on for Windows) failed to extract English fields like `TargetUserName` or `IpAddress`, resulting in empty statistics tables.

* **Raw Log Analysis:** Analyzing the raw XML payloads revealed the localized field names generated by the OS:
    * `IpAddress` -> `Adres źródłowy sieci`
    * `TargetUserName` -> `Nazwa_konta`

### Final Detection Logic (SPL)
Using the exact extracted fields, the following Splunk Search Processing Language (SPL) query was constructed to aggregate the failed attempts:

    index=main EventCode=4625
    | stats count by Nazwa_konta, "Adres źródłowy sieci", ComputerName
    | sort - count

### Evidence
* **Result:** Successfully identified the attacker's IP (`192.168.10.22`) and the compromised account with over 250 failed attempts in a matter of seconds.

### Creating a SOC Tier 1 Dashboard (Risk Scoring)
To shift from raw logs to actionable intelligence, a custom Dashboard was created. Using Splunk's `eval` function, a dynamic **Risk Level** was assigned to IPs based on the volume of authentication failures. 

* The Dashboard provides an immediate visual queue (`CRITICAL 🚨`, `HIGH 🔴`) for Tier 1 Analysts to begin triaging the event.
